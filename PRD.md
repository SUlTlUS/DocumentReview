## 1. 产品概述

### 1.1 产品定位

基于 RAG（检索增强生成）技术的智能文档审核平台，帮助用户快速发现合同、协议等文档中的风险条款、合规问题和权益隐患。

### 1.2 核心价值

- **上传即解析**：支持 PDF、DOCX、TXT 上传，自动提取文本并建立 RAG 索引
- **AI 智能审核**：从合规风险、条款缺失、表述模糊、权益不对等四个维度自动分析文档
- **文档问答**：基于 RAG 检索，用户可针对文档内容随时提问，获取带引用的回答
- **轻量部署**：无需 GPU，TF-IDF 向量化 + DeepSeek API 即可运行

### 1.3 目标用户

法务、HR、采购、行政人员，以及需要对合同/协议进行快速风险审查的非专业人员。

### 1.4 功能优先级

| 优先级 | 功能 | 理由 |
|--------|------|------|
| **Must Have** | 文档上传与解析、四维度智能审核、审核结果展示 | 没有这些不能叫"文档审核系统" |
| **Must Have** | 文档列表管理、文档删除 | 基础 CRUD 闭环 |
| **Should Have** | RAG 文档问答 | 差异化竞争力，但审核主流程不依赖它 |
| **Could Have** | Markdown 渲染、建议问题快捷按钮 | 体验优化，不影响核心功能 |
| **Won't Have (v1.1+)** | 用户登录/权限、审核报告导出、批量对比 | 非 MVP 范围，后续迭代 |

## 2. 功能需求

### 2.1 文档上传（Upload）

**功能描述**：用户拖拽或选择本地文件上传，系统自动解析文本并建立 RAG 索引。

**支持格式**：PDF、DOCX、TXT

**处理流程**：
1. 用户拖拽文件到上传区域（或点击选择）
2. 前端校验文件类型（仅 PDF/DOCX/TXT）
3. 上传文件到 `/api/documents/upload`（multipart/form-data）
4. 后端：保存文件 → 解析文本（parser.py）→ TF-IDF 分块索引（rag.py）→ 写入 DB
5. 前端显示上传结果，可跳转至审核页

**状态标记**：
- `uploaded`：文件已保存
- `parsing`：正在解析
- `ready`：解析完成，索引就绪
- `parse_failed`：解析失败

**用户故事**：

> 作为**法务专员**，我希望**拖拽合同文件到页面上就能自动上传并解析**，以便**省去手动输入文件信息的麻烦，快速开始审核工作**。

**验收标准**：

- Given 用户打开上传页面
- When 拖拽一个 PDF/DOCX/TXT 文件到上传区域
- Then 文件上传成功，页面显示"已索引 N 个文本块"，文档状态变为 ready
- When 上传一个非支持格式的文件（如 .xlsx）
- Then 前端提示"仅支持 PDF/DOCX/TXT 格式"，文件不会上传

---

### 2.2 文档管理（Document List）

**功能描述**：展示所有已上传文档列表，支持查看状态、发起审核、进入问答、删除文档。

**列表字段**：文件名、文件大小、上传时间、内容摘要、审核状态、操作按钮

**状态展示**：
- 待审核（status=ready, review_status=pending）
- 审核中（review_status=reviewing）
- 已审核（review_status=completed）
- 解析失败（status=parse_failed）

**操作按钮**：
- 审核：跳转至 ReviewResult 页面
- 问答：跳转至 Chat 页面
- 删除：确认后删除文档及所有关联数据

**用户故事**：

> 作为**法务专员**，我希望**在一个列表页查看所有已上传文档的状态和审核进度**，以便**快速判断哪些合同需要优先处理**。

**验收标准**：

- Given 系统中已有 5 份文档（状态各不相同）
- When 用户进入文档列表页
- Then 5 份文档按上传时间降序排列，每份显示文件名、大小、上传时间、解析状态、审核状态
- When 用户点击某份文档的"删除"按钮并确认
- Then 该文档及其关联的审核记录、审核项、问答数据全部删除，列表刷新

---

### 2.3 文档审核（Review Result）

**功能描述**：调用 DeepSeek LLM 对文档进行风险分析，从四个维度返回结构化审核结果。

**审核维度**：
| 维度 | 说明 |
|------|------|
| 合规风险 | 违反法律法规、行业标准的条款 |
| 条款缺失 | 缺少保密、违约责任、争议解决等保护性条款 |
| 表述模糊 | 含义不清晰、容易产生歧义的表述 |
| 权益不对等 | 权利义务明显不对等的条款 |

**风险等级**：高（high）、中（medium）、低（low）

**审核结果字段**：
- `summary`：审核总结（1-2句话）
- `items`：风险项列表，每项包含 category、severity、title、description、suggestion、source_text

**展示内容**：
- 审核时间、总问题数、高风险数
- 按严重程度排序的风险项卡片（左侧彩色边框区分等级）
- 每个风险项：等级标签 + 分类标签 + 问题描述 + 原文片段 + 修改建议

**可重复审核**：支持对同一文档重新发起审核，生成新的审核记录。

**用户故事**：

> 作为**法务专员**，我希望**系统自动从合规风险、条款缺失、表述模糊、权益不对等四个维度审查合同并标注风险等级**，以便**优先关注高风险条款，而不是逐页逐条翻看**。

**验收标准**：

- Given 一份已解析完毕的合同文档（status=ready）
- When 用户点击"审核"按钮
- Then 20-30 秒后返回结构化审核结果，包含 summary 和 items 列表
- And items 中每个风险项包含：category、severity、title、description、suggestion、source_text
- And 同一文档可重复审核，每次生成一条新的 Review 记录，历史记录保留

---

### 2.4 文档问答（Chat Q&A）

**功能描述**：基于 RAG 检索，用户针对文档内容提问，AI 返回引用了原文的回答。

**技术流程**：
1. 用户输入问题
2. TF-IDF 检索（rag.py retrieve_context）找到最相关的 5 个文本块
3. 将相关文本块作为 context，携带对话历史，发给 DeepSeek LLM
4. 返回回答，前端渲染 Markdown

**功能细节**：
- 首次进入时显示 4 个建议问题快捷按钮
- 对话历史在当前会话内持续保留（取最近 4 轮）
- AI 回答支持 Markdown 渲染
- 加载中显示三点动画

**用户故事**：

> 作为**法务专员**，我希望**针对合同内容随时提问（如"违约责任怎么约定的？"）并获得引用原文的回答**，以便**不用逐页翻找，快速定位关键条款**。

**验收标准**：

- Given 一份已索引的文档
- When 用户进入问答页
- Then 页面显示 4 个建议问题快捷按钮
- When 用户输入问题并发送
- Then 15 秒内返回 AI 回答，回答中引用文档原文
- And 对话历史在会话内持续保留（最近 4 轮）

## 3. 非功能性需求

### 3.1 性能要求

- 文档上传 + 解析 + 索引：≤ 10 秒（小于 1MB 文本）
- 文档审核（DeepSeek API 调用）：≤ 30 秒
- 问答响应：≤ 15 秒
- 前端页面加载：首屏 ≤ 2 秒

### 3.2 前端样式规范

- 主题：深色主题，背景 #0D1117
- 强调色：#58A6FF（蓝色）
- 卡片背景：#161B22
- 边框色：#30363D
- 字体：系统默认无衬线字体
- 布局：固定左侧导航栏（240px），主内容区 margin-left: 240px

### 3.3 环境依赖

- Python 3.10+
- Node.js（前端构建）
- httpx < 0.28（openai SDK 兼容要求）
- DeepSeek API Key（需配置在 backend/.env）
