# Day3 v1.0 / v2.0 对比报告

## 测试方法

使用同一份 `backend/tests/fixtures/test_contract.txt`，分别运行 Day2 内联 Mock 规则和 Day3 LCEL Mock 管线，每个版本连续执行三次。该测试用于比较代码路径、结构、输出稳定性与本地编排开销，不用于推断 DeepSeek 的响应速度或法律判断能力。复现命令为 `backend\.venv\Scripts\python.exe scripts\run_v1_v2_comparison.py`。

| 指标 | v1.0 inline Mock | v2.0 LCEL Mock |
|---|---:|---:|
| 重复次数 | 3 | 3 |
| 最短耗时 | 0.001ms | 1.585ms |
| 平均耗时 | 0.001ms | 12.126ms |
| 最长耗时 | 0.003ms | 32.663ms |
| 风险项 | 3 | 3 |
| 高/中/低 | 1 / 2 / 0 | 1 / 2 / 0 |
| 默认维度 | 代码内四维 | YAML 五维 |
| 阶段可追踪 | 否 | 结构、维度、汇总 |

两条路径都稳定发现“违约责任仅约束乙方”“合理期限”和“缺少保密条款”，说明升级没有牺牲既有核心行为。v2.0 会把高风险放在前面，并在数据库记录 `pipeline_version=v2.0`、`prompt_version=2.0`。结果文字中保留 Mock 声明，防止用户误认为来自真实模型。

## 工程差异与解释

v1.0 的绝对耗时更低，这是预期结果：它只是单函数字符串匹配，几乎没有调度成本。v2.0 即使在 Mock 下也会构造五个 Runnable、在线程池中按最大并发 2 执行、记录三个阶段耗时，并进行 Pydantic 校验和确定性去重，因此本地多出约毫秒级开销。这个数据不能用于声称 v2.0 “更快”；它证明的是在真实 LLM 调用占据主要耗时之前，编排本身的固定成本很小且可测量。真实模型环境下是否因并行得到总时延收益，必须单独压测。

可维护性是差异最大的部分。v1.0 新增审核维度要修改 Prompt 字符串、分类校验，通常还会联动展示逻辑；v2.0 的并行节点直接从 YAML 生成。知识产权实验只增加 `name/key/description` 三项配置，六维测试立即通过，随后又用独立提交恢复五维默认。这是可审计的“配置驱动”证据，不是架构图上的主观描述。

可观测性方面，v1.0 只知道一次调用成功或失败，难以判断问题来自合同理解、某个维度还是报告汇总。v2.0 对 structure、dimensions、report 分别记录耗时和计数，JSON 解析失败又不会泄露原始合同或回答。OCR 同样从“空 PDF 直接失败”升级为明确的 digital → OCR fallback 状态机。

本次对比仍有清晰限制。v1 基线函数和 v2 管线均为确定性 Mock，能证明接口契约、流程稳定性、排序和错误边界，不能证明法务准确率。真实 DeepSeek 需要经匿名化合同集、法务双人标注、召回率/误报率和成本测试；真实 PaddleOCR 需要扫描清晰度、旋转、印章遮挡和多页表格样本。当前外部能力统一结论为 `PASS (Mock)`。
