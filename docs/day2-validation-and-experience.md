# Day2 验证与体验报告

## 结论

Day2 已形成可运行、可测试、可回退的 v1.0 基线。系统完成 PDF、DOCX、TXT 三种合同上传，建立文档级持久化索引，支持四维合同审核与基于原文引用的连续问答。后端 17 项测试、前端 4 项单元测试、TypeScript 类型检查、生产构建及 Edge 真实浏览器主流程均通过。外部大模型与 OCR 在本阶段没有接入生产凭据，因此对应结论明确记为 `PASS (Mock)`，不将编排通过误写为真实模型质量通过。

## 上午知识清单

### 三个应用场景与用户故事

1. 法务初筛：作为企业法务，我希望批量上传采购合同并快速定位责任失衡、表述模糊和条款缺失，以便把人工时间投入到高风险段落。
2. 业务自查：作为采购负责人，我希望用自然语言询问付款、验收和违约责任，并看到回答所依据的原文片段，以便在送审前修正文档。
3. 审核留痕：作为合规负责人，我希望保留每次审核结果和问答记录，同时可删除文档及全部关联数据，以便兼顾追溯与数据最小化。

模型选择按“质量、中文合同能力、部署约束、成本”四项比较：DeepSeek 适合复杂推理和云端快速接入，Qwen3 兼顾中文表现与本地部署生态，Llama 更适合已有英文或跨语种私有化基础设施的团队。云模型前期运维成本低但涉及按量费用、网络和数据出境评估；本地模型固定资源投入更高，却能强化数据控制和稳定吞吐。本项目 v1.0 使用可替换 Mock Provider 验证接口和状态流，模型质量结论为 `PASS (Mock)`。OCR 方案同理：云 OCR 对复杂版式通常开箱即用，本地 PaddleOCR 更便于数据闭环；Day2 只为 Day3 留出解析边界，结论为 `PASS (Mock)`。

## 下午实现与证据

数据层创建 documents、document_chunks、review_results、review_issues、chat_messages 五张表，关联记录使用外键和级联清理。FastAPI 暴露健康检查、文档上传/列表/详情/删除、审核创建/读取、问答创建/历史共 9 个端点。前端提供上传、文档列表、审核结果和问答四个路由；桌面端使用 240px 侧栏，移动端转换为顶部导航。标题、风险指标和等待态采用轻量 Kinetic Typography，仅使用 opacity/transform，时长控制在 150–300ms，并在 `prefers-reduced-motion` 下提供静态回退。

真实格式验证使用同一份采购服务合同生成 TXT、DOCX 和嵌入中文字体的 PDF。三种文件均通过解析器关键短语断言和真实上传 API 参数化测试。PDF 使用 Poppler 渲染后人工检查，未发现乱码、截断或溢出；DOCX 通过 OOXML 结构、段落提取和 API 全链路校验。本机缺少 LibreOffice，Word COM 等价导出无响应，因此 DOCX 没有宣称完成视觉渲染，避免把环境能力缺口包装成通过。

浏览器验证使用 Microsoft Edge 完成“上传 TXT → 发起审核 → 查看风险 → 提问 → 查看引用”的实际交互；全新会话控制台为 0 error、0 warning，并验证 375px 移动视口。安全检查覆盖扩展名白名单、空文件拒绝、数据库参数化访问、上传文件隔离和敏感配置忽略；生产依赖审计为 0 个已知漏洞。v1.0 不承诺恶意文档沙箱、真实模型提示注入防护或企业级鉴权，这些属于后续上线门槛。

## 体验总结

本次开发最重要的收获是把“能返回结果”与“结果值得信任”分开验证。上传体验如果只显示成功提示，用户无法判断文档是否真正进入索引；因此界面同时呈现状态、文件类型和后续动作。审核页不采用连续长文本，而是先给出风险总览，再按维度组织问题、证据和建议，让法务能够先决定阅读顺序。问答页把引用片段与回答放在同一上下文中，降低模型回答脱离合同原文的风险。

视觉上采用深色 Trust & Authority 方向，强调稳定、专业和信息层级。Kinetic Typography 没有作为装饰铺满页面，而是用于页面标题进入、关键数字反馈和处理中状态，既建立节奏也避免干扰长文本阅读。移动端真实验证暴露过焦点跳转导致页面轻微偏移的问题，现已通过 `preventScroll` 和跳转链接焦点样式修正。这说明响应式验收不能只依赖桌面缩放。

工程上，小步提交使数据库、解析、API、页面和测试都能独立定位与回退；Mock Provider 则让没有外部密钥的环境仍可验证稳定契约。它的边界也必须清楚：Mock 能证明结构化输出、错误处理和页面状态，却不能证明模型对真实合同的法律判断质量。Day3 应在保持 9 个端点兼容的前提下，把 OCR、YAML Prompt、结构化校验、并行审核和运行日志升级为明确分层的编排系统，再用同一测试合同对比 v1/v2，而不是仅比较主观措辞。
